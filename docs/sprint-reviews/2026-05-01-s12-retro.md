## Architectural lessons — Sprint 12 retro

The user enumerated five lessons in the dogfood brief; each is captured here with
context, impact, and proposed Sprint 14 mitigation.

### 1. Multiple parallel `Literal` types for the same concept

**Symptom:** `PaymentMode` was independently declared in 4 places — `gecko_core.models`,
`gecko_core.sessions.models`, `gecko_api.api_models`, and the SQL CHECK constraint
on `sessions.payment_mode`. Adding `'cdp'` for Sprint 12 Track C required edits to
all four; missing any one produced a runtime mismatch that a user-flow test caught
late.

**Impact:** S12.5-TEST-01..04 was an unplanned mini-sprint to back-fill schema-drift
guards; ~10 commits of cleanup; CDP listing readiness slipped by ~2 days.

**Mitigation (folded into S14):** the **S14 hardening track** adopts the
S13-PAY-01 X402Client Protocol shape as the pattern: one canonical declaration
(in `gecko_core.payments.protocol`), every consumer imports from it. Schema
generation (Pydantic JSON Schema → Supabase migration draft) becomes a Sprint 15
ticket — not in S14 scope, but the constraint is named.

### 2. Track A's stubbed payload took ~10 fix iterations

**Symptom:** `S12-CDP-01..03` plus 10 follow-on fixes (EIP-3009 buyer signing,
USDC EIP-712 domain, ResourceInfo, assetTransferMethod, Ed25519 key parsing,
PEM unescape, base+sepolia registration, payTo on eip155:* routes, etc.) — the
CDPX402Client landed and immediately took two business days of incremental
hardening before the live smoke passed.

**Impact:** the live mainnet smoke (S12-LIST-01) gated on Track A-like stability,
so this delayed the listing milestone. None of the 10 fixes were architectural —
all were "the CDP wire format requires X but the docs underplayed it."

**Mitigation (folded into S14):** the **3 latent CDP issues** in the carry-forward
list (max_timeout_seconds, /.well-known/x402 extra advertisement, payTo checksum)
are the residual long-tail of this same iteration cycle. S14-CDP-HARDEN-01..03
land them. Going forward, every new facilitator gets a **conformance test suite
applied before the first live call**, derived from S12.5-TEST-04's gated /verify
integration test pattern. Captured in the build plan.

### 3. Permit2 vs eip3009 dispatch confusion

**Symptom:** the CDP facilitator accepts both Permit2 and EIP-3009 transfer methods;
Sprint 12's first live attempt sent Permit2 by default which Coinbase quietly
rejected. The fix (`fix(cdp): set assetTransferMethod=eip3009 in
PaymentRequirements.extra`) was a one-line edit but required a debug round to
discover.

**Impact:** ~half a day debugging an opaque rejection; no log surface from CDP
named "wrong transferMethod"; we inferred it from header inspection.

**Mitigation:** Sprint 13 Track C (X402Client Protocol) already names this:
`assetTransferMethod` becomes a per-client capability flag, not implicit. **S14
will not re-introduce this** — it ships once S13-PAY-01 lands and is observable
in the `bb doctor` output (per F-S14 finding "facilitator capabilities surface").

### 4. payTo network mismatch — Solana address advertised on Base routes

**Symptom:** `/.well-known/x402` advertised the Solana wallet address as `payTo`
for Base-mainnet routes. Bazaar quoted the listing without an immediate visible
rejection because the route's chain/asset combination implicitly invalidated it,
but a direct CDP call would have rejected.

**Impact:** caught in pre-listing review (`S12-LIST-01` runbook), fixed by
`fix(api): use GECKO_WALLET_ADDRESS_BASE as payTo on eip155:* routes`. **No
production exposure**, but it's a representative drift bug.

**Mitigation:** S14-CDP-HARDEN-02 owns the /.well-known/x402 advertisement
contract: routes only advertise networks where the resolver registry has a
matching wallet, not the union of all configured wallets. Add a startup-time
assertion that fails fast if any route's `payTo` doesn't pass the network's
address-format check.

### 5. Recommendation: Sprint 13 S13-PAY-01 (X402Client Protocol) is the right shape

**The four lessons above all reduce to: the seam wasn't narrow enough.** Each
facilitator's quirks (CDP wants eip3009, Solana wants Permit2 surrogate, frames.ag
proxies through its own bridge, the future Cloudflare slot uses HTTP signatures)
should be **invisible to the caller** — every call goes through one Protocol
method and the client knows its own capabilities.

S13-PAY-01 already exists on disk (the in-flight files in `git status` confirm
`payments/protocol.py` and `payments/factory.py` are written). The Sprint 14
plan must **ensure all consumers go through this seam, not just new ones.**
Concretely:

- `gecko_api` payment router: migrate to `resolve_client(intent)` factory, drop
  the inline if/elif on `X402_NETWORK`.
- `apps/cli` `bb doctor` payments section: read from `client.facilitator_id` +
  `client.supported_networks` instead of re-deriving from env.
- `gecko_mcp` MCP tools: any wallet/payment introspection goes through the
  Protocol, not a bespoke import.

**Action item:** Sprint 14's hardening track (S14-PAY-MIGRATE-01..03) is the
"every consumer goes through the seam" pass. Land it before any new
facilitator-shaped work in S15 (Cloudflare).
